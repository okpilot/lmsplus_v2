## Durable knowledge

- **When a function has a safety-critical ordering invariant (A must happen before B, or a downstream handler fires with wrong state), test the invariant directly by capturing the value of the shared variable INSIDE the B mock — not just the final state.** The canonical pattern: a spy on `stop()` captures `recorder.onstop` at call time; a null value proves `onstop` was nulled before `stop()` fired. Final-state assertions are insufficient because a later reset (`setState(INITIAL_RECORDER_STATE)`) can mask the brief wrong state. First confirmed: `teardownCapture` onstop-before-stop ordering, `audio-recorder-core.test.ts` (2026-07-02).
- **Bound `durationMs` (or any `Date.now() - ref.current` computation) with a tight ceiling test (`< 10_000 ms`).** If the `startedAt` ref is never set (stays 0), `Date.now() - 0 ≈ 1.75 × 10¹²` — wrong but passes a `>= 0` check. The upper bound catches the "never-set startedAt" regression. First confirmed: `use-audio-recorder.test.ts` (2026-07-02).
- **When a catch block has two sequential effects gated by different conditions — teardown (`if (flagCreated) await cleanup()`) then a specific-error branch (`if (err.message.includes('token')) return specificMsg`) — test the specific-error path WITH the combination assertion** (assert teardown was called AND the specific message is returned). Testing each effect separately leaves an invisible regression: if someone moves the early-return before the teardown, both individual tests still pass but the teardown is silently skipped. First confirmed: `study.ts` `active_exam_session` + `endDiscovery` ordering, PR #1011 sweep.
- **Bash security-gate parsers need four extra cases beyond happy/unhappy paths:** (a) empty input — fail closed; (b) both tokens present — the blocking token must win regardless of order; (c) CRLF line endings — `[[:space:]]` absorbs `\r`, pin the behaviour so a future `[ \t]` substitution is visible; (d) leading-whitespace variant of each token — if the regex allows indented tokens (as this parser does for both BLOCKED and APPROVED), add a pinning test so intent is documented. Use a `run_case_raw_bytes` helper with `printf '%b'` for the CRLF case. Pattern confirmed: `.claude/hooks/run-security-auditor.test.sh` (2026-06-10, issue #832).
- **Every cleanup/restore mutation in generated tests must destructure `{ error }`** (finally blocks, afterEach, afterAll included) per code-style.md §5 — the cleanup path is not exempt; log via `console.error`, never throw inside `finally` (biome noUnsafeFinally). Promoted at count=2 (13fa0249, 50c81b94 — both finally-block restore UPDATEs caught by the orchestrator pre-commit).
- **Always `vi.hoisted()` for any variable a `vi.mock()` factory references** — no exceptions. Vitest hoists factories above plain `const`s, so the var is `undefined` at factory time and the mock silently no-ops. Bug is invisible until the test runs.
- **jsdom: pre-hydration state is untestable.** RTL's `render()` runs inside `act()`, which flushes effects, so a hydration guard's pre-hydration branch (skeleton/disabled) never appears. Test only the post-hydration state — not a missing test, a jsdom constraint.
- **Co-locate tests, one file per source file, behavior-first names; new `_hooks/`, `_utils/`, `lib/` files ship with a test in the same commit** (per `code-style.md` §7). Disallowed `it(...)` title patterns (impl leakage) and required router-URL / lifecycle / reload assertions are all in §7 — defer to it, don't re-document.
- **`buildChain` is defined locally per test file** (Proxy that forwards chained methods and resolves to a fixture); never import it across files. Sequence multi-call queries with a per-call queue or a **per-table** call counter — never a global counter (microtask ordering in `Promise.all` defeats array-index assumptions).
- **Mock the right Supabase surface:** `@repo/db/client` (browser), `@repo/db/server` (Server Component/route), `@repo/db/admin` singleton (mock whole module — created at import time), raw `@supabase/supabase-js` (intercept before top-level `SERVICE_ROLE_KEY` checks throw), `@repo/db/middleware` (plain `{ status, headers }`, not `NextResponse.next()`).
- **Always test the non-array RPC branch** when production has `Array.isArray(data) ? data : []` or `data ?? []` — `{ data: null, error: null }` is a real gap reviewers flag as dead code. Read the full loop→map→filter pipeline first: a null-data empty Map can drop filtered subjects entirely, not just zero their fields.
- **`sessionStorage` must be replaced via `Object.defineProperty(globalThis, 'sessionStorage', { value: {...}, writable: true })` in `beforeEach`** — `vi.spyOn(window.sessionStorage, 'setItem')` does not intercept calls made inside the module under test in jsdom. Hoist the mock fns (`mockSessionStorageSetItem` etc.) with `vi.hoisted()` so the factory can reference them. Pattern confirmed: `use-exam-start.test.ts` and `quiz-recovery-handlers.test.ts`.
- **Stub `server-only` via a `vitest.config.ts` alias** (`vi.mock('server-only')` fails — Vite import-analysis throws first). One-time config change.
- **Re-throw `isRedirectError` tests:** Server-Component content wrappers must verify happy path, regular-error fallback, AND redirect re-throw (mock `next/dist/client/components/redirect-error`).
- **Confirm the exact production fallback string before asserting it** in try/catch wrapper tests; `vi.spyOn(console,'error')` to silence + assert logging.
- **Async Server Component layouts (layout.tsx) ARE testable:** call `await Layout({ children })` directly to get the JSX, then `render(jsx)`. Mock `next/navigation` for `redirect` (vi.hoisted); mock `'use client'` children to plain `<div data-testid="…" data-prop="…">` stubs so RTL can run without hook deps. Wrap `AppLayout({ children })` calls in try/catch when `redirect` is mocked as a no-op — in real Next.js it throws, so the try/catch future-proofs the test against the real throw path without needing `isRedirectError`. Pattern first confirmed: `app/app/layout.test.tsx` (2026-06-01).
- **"no error banner" tests must also assert the positive content renders.** A test that only asserts `queryByRole('alert') === null` on a success path is half-tested: it doesn't catch the regression of accidentally wrapping the content in `{!loadFailed && ...}`. Pair `queryByRole('alert') toBeNull` with `getByTestId('content-component') toBeInTheDocument` in the same test or add a sibling test that asserts both. Pattern applies to any `{flag && <ErrorBanner>}` + `<Content>` layout (PR bac94f2a).
- **Integration tests for SECURITY DEFINER RPCs must apply `supabase db push --local` first.** The schema cache only updates after migrations are applied; `PGRST202` (function not in schema cache) is the symptom. Use `supabase migration repair --status reverted <id>` to unblock a push when a prior remote migration is missing locally. `getAuthenticatedClient()` from `__integration__/setup.ts` provides the per-role authenticated client needed to test auth.uid()-dependent SECURITY DEFINER functions — service-role calls always get `auth.uid() = null`. Cross-org rejection tests are non-vacuous only when a `.single()` lookup confirms the target row actually exists before asserting the RPC rejects it. Pattern confirmed: `rpc-record-auth-event.integration.test.ts` (2026-06-06). **If a migration was edited IN-PLACE after its first `db push`, `db push` no-ops (its hash is already in the ledger) and the local function diverges from the repo — run `supabase db reset` + re-seed instead. See CLAUDE.md § Local migrations (#794, count=2).**
- **Pure DB trigger coverage goes in `packages/db/src/__integration__/`, not unit tests.** The harness (`setup.ts`, `seed.ts`, `cleanup.ts`) creates real orgs/users/questions and runs real RPCs. Naming: `trigger-*.integration.test.ts`. Run: `pnpm --filter @repo/db test:integration`. Migration must be applied first (`npx supabase db push --local`); confirm with `docker exec supabase_db_lmsplusv2 psql -U postgres -c "SELECT routine_name FROM information_schema.routines WHERE routine_name LIKE '%<name>%';"`. First confirmed: `trigger-stamp-last-active.integration.test.ts` (2026-06-06).
- **TIMESTAMPTZ comparison in integration tests: parse, never string-compare.** PostgreSQL serialises TIMESTAMPTZ as `+00:00`; JS `toISOString()` produces `Z`. String equality fails. Always compare via `new Date(value).getTime()` on both sides.
- **Playwright helper files with unexported pure logic warrant a co-located Vitest unit test.** When a `helpers/*.ts` file inside `e2e/` contains pure, edge-case-prone logic (base64 decode/encode, chunk-split/reconstruct) alongside Playwright-coupled async functions, export the pure helpers and add a `.test.ts` next to the file. `Buffer` is available in the jsdom environment (Node global). Do NOT import `@playwright/test` types into the test file — define a minimal local fixture type for the cookie shape the helper accepts. Pattern first confirmed: `force-token-refresh.ts` (2026-06-07).
- **CHECK-constraint rejection tests use service-role admin INSERT, not RPCs.** A DB-level `CHECK` fires on raw row writes — no plpgsql body involved, so execution-time deferral (the ON CONFLICT 42P10 problem) does NOT apply. Test the CHECK by inserting a violating row via the `admin` (service-role) client and asserting the returned error contains `23514` / `check` / the constraint name. Always add a positive control asserting a valid row inserts without error (especially for the branch that legitimately uses the field, e.g., `short_answer` with non-empty `accepted_synonyms`). Pattern first confirmed: `rpc-vfr-rt-constraint-regression.integration.test.ts` mig-094 block (2026-06-10).
- **PostgREST unnamed TEXT parameter encoding:** PostgREST (14.12) maps the JSON key `""` (empty string) to the first unnamed `text` parameter of a SQL function. Call with `body: JSON.stringify({ '': input })`. Scalar results are returned in PostgreSQL anonymous-row text format: `{value}` (no spaces for normal strings), `{ value }` when the input had leading/trailing whitespace, `{}` for empty string. Unwrap with `raw.slice(1, -1).trim()`. CRITICAL LIMITATION: when the input contains `"` characters, `JSON.stringify({'': '"cleared"'})` produces `{"":"\"cleared\""}`, and PostgREST passes the raw JSON-encoded bytes (backslash-escape sequences included) to the SQL function as the TEXT arg. The function receives `\"cleared\"` (with actual `\` chars), NOT `"cleared"`. Do NOT write `it.each` test cases with `"` in the input when using this calling convention — those test cases will fail with `\cleared\` instead of `cleared`. Verified via psql: the SQL function itself correctly strips `"` when called directly. Pattern first confirmed: `rpc-vfr-rt-start.integration.test.ts` `normalize_answer` tests (2026-06-10).
- **Partial unique index "permit after ended/deleted" must be an explicit named test.** When a partial unique index guards active rows (`WHERE ended_at IS NULL AND deleted_at IS NULL`), the "index does NOT block a new row after the prior one is ended/soft-deleted" property is load-bearing (students must be able to re-take). This is easy to cover implicitly (fixture setup calls start twice, second succeeds because first session was submitted) but that is insufficient: a named `it('allows a new session after the previous one is ended')` test actually catches a regression. Pattern first confirmed: `uq_vfr_rt_exam_session_active` (mig 096) / `rpc-vfr-rt-start.integration.test.ts` (2026-06-10).
- **`isContentEditable` is `undefined` in jsdom** — the `HTMLElement.isContentEditable` DOM property is not implemented; setting `el.contentEditable = 'true'` does NOT make `el.isContentEditable` truthy. Production code that reads `el.isContentEditable` will return `undefined` (falsy) in tests. Assert with `toBeFalsy()` for the non-contenteditable path; drop the contentEditable-specific test case or note the jsdom limitation. Also: JavaScript `||` chains return the last evaluated value, not a boolean — `el.tagName === 'INPUT' || el.tagName === 'TEXTAREA' || el.isContentEditable` returns `undefined` (not `false`) when all three are falsy. Assert with `toBeFalsy()`, not `toBe(false)`. First confirmed: `quiz-key-actions.test.ts` (2026-06-14).
- **`window.dispatchEvent(new KeyboardEvent('keydown', { key }))` triggers window listeners in jsdom** — wrap in `act(...)` to flush state updates from the listener. To simulate a typing target, create a real DOM element (`document.createElement('input')`), append to `document.body`, then pass the element as the `target` override via `Object.defineProperty(event, 'target', { value: el })` — `KeyboardEvent` constructor ignores the `target` init option. Pattern confirmed: `use-quiz-keyboard.test.ts` (2026-06-14).
- **Do NOT write `Promise.all` concurrency tests for PL/pgSQL EXCEPTION paths.** Two parallel RPC calls via `Promise.all` are NOT guaranteed to overlap at the INSERT step — OS scheduling and Postgres connection timing are non-deterministic. The EXCEPTION path is deterministically covered by: (1) a direct service-role INSERT of a duplicate row (proves the index rejects it with 23505), plus (2) an idempotent-resume test (proves the RPC's step-5 SELECT path). Those two together are the correct non-flaky proxy. Pattern first confirmed: `uq_vfr_rt_exam_session_active` EXCEPTION handler analysis (2026-06-10).

- **Testing a `Promise.race` timeout branch requires fake timers + `vi.runAllTimersAsync()`.** When production uses `Promise.race([realWork, new Promise(resolve => setTimeout(resolve, N))])`, the never-resolving mock (`mockFn.mockReturnValue(new Promise(() => {}))`) hangs the test unless you advance time. Start the call without awaiting it, call `await vi.runAllTimersAsync()` to drain all pending timers, then await the pending promise. `vi.advanceTimersByTime(N+1)` is insufficient when the race contains microtask chains — `runAllTimersAsync` is the safe choice. Isolate in `vi.useFakeTimers()` / `vi.useRealTimers()` try/finally per test. First confirmed: `submitQuizSession` DRAFT_CLEANUP_TIMEOUT_MS branch, `quiz-submit.test.ts` (2026-06-21, #909).
- **Assert per-item result shape in array-mapping actions, not just length.** When a Server Action maps a flat RPC array (snake_case wire keys → camelCase result keys) and returns the array to the caller, `toHaveLength(N)` passes even when every item has all fields `undefined` due to a key-name regression (e.g., `r.question_id` misspelled as `r.questionId`). Add a per-item shape assertion that checks `typeof r.questionId`, `typeof r.isCorrect`, and the literal value of any constant field (e.g., `r.correctOptionId === 'b'` for seeded questions). First confirmed: `batchSubmitQuiz` results mapping, `batch-submit.integration.test.ts` (2026-06-20, #925 Phase 2 gap).
- **Test the `ended_at IS NULL` session guard in actions that own it.** Any action that queries `quiz_sessions` with `.is('ended_at', null)` must have a test that first ends a session (via `completeQuiz` or `batchSubmitQuiz`) then calls the action on it. `nonexistent UUID` and `cross-user session` tests do NOT cover this guard — they trigger missing-row rejections, not the active-session filter. The test is: `seedOpenSession → signInAs → completeQuiz(sessionId) → action(sessionId) → expect error`. First confirmed: `checkAnswer` ended-session guard, `check-answer.integration.test.ts` (2026-06-20, #925 Phase 2 gap).
- **App-layer integration tests for Server Actions (the #925 tier) follow the quiz-report pattern:** per-file `const suffix = Date.now()`, `const admin = getAdminClient()`, `beforeAll` seeds org → users → refs → questions → `getAuthenticatedClient` → open/completed sessions (DO NOT call `signInAs` in `beforeAll` — the cookie jar is reset per-test by `vitest.integration.setup.ts`). Call `signInAs(email, password)` at the TOP of EACH test. `afterAll` uses the error-accumulator pattern (`const errors: string[] = []`; `try { cleanupTestData } catch push; if errors.length===0 try { cleanupReferenceData } catch push; if errors.length throw`). Every action that ENDS a session (`completeQuiz`, `batchSubmitQuiz`) needs a FRESH `seedOpenSession` per test — do not share a completed session across tests. checkAnswer does NOT end the session, so a shared `seedOpenSession` in `beforeAll` is safe for that file. Array index access on `questionIds[]` is typed `string | undefined` — guard with a null check and throw if falsy. Variables captured but only referenced inside `signInAs` (e.g., `studentBClient` in `completeQuiz` tests where B is only identified by email) must be omitted or prefixed with `_` — Biome flags `noUnusedVariables`. First confirmed: `{start,submit,check-answer,complete,batch-submit}.integration.test.ts` (2026-06-20, #925 Phase 2).

- **Splitting a test file is a faithful MOVE — the moved `describe` block must carry ALL of the source's sentinel-var declarations + `if(sentinel)` cleanup guards + every branch, not just the happy path.** It is easy to silently drop a defensive guard during the extraction. Before committing a split, re-diff the moved block's setup/`beforeAll`/`afterAll` against the source. Count=2: #698/#666 dashboard-query split dropped error-path/edge-case branches; #951 mig-125 split dropped the `if(orgId)` guard on the `afterAll` `cleanupTestData` call (both caught pre-commit by impl-critic). Also carry the §7 multi-step-cleanup shape intact (error-accumulator + `finally { ids.length = 0 }` reset + per-step `errors.length === 0`-gated try/catch).

- **`it.each` tuple column order must match parameter destructuring AND the title template simultaneously.** The `%s` substitution in a `it.each` title is positional — `%s %s` with `['a', 'b']` produces title "a b". If you swap the tuple order without swapping the parameter names (or vice versa), the title reads backwards AND the assertion tests the wrong value. The canonical fix: put `[expected, input]` (result first) and destructure `(expected, input)` so title says "returns expected for input" and the assertion is `expect(fn(input)).toBe(expected)`. Read the title aloud after writing to catch the inversion. First confirmed: `audioExt` it.each in `submit-section-response-helpers.test.ts` (2026-07-02) — `[fileName, expected]` tuple with `(expected, fileName)` params silently inverted every assertion.
- **Supabase Storage mock shape:** when mocking a client used for both DB queries AND file storage, the mock factory must include `storage: { from: mockStorageFrom }` (alongside `from: mockFrom`, `auth: { getUser }`) where `mockStorageFrom` is a `vi.fn()` that returns `{ upload: vi.fn(), remove: vi.fn() }`. The `mockStorageFrom` ref itself must be hoisted via `vi.hoisted()` so tests can call `mockStorageFrom.mockReturnValue(...)` per-test. Pattern confirmed: `submit-section-response.test.ts` (2026-07-02).
- **For boolean RPC results where `false` is a valid success, test the `false` case explicitly.** When production checks `data === null` (not `!data`) to detect failure, `data = false` is treated as success. Without a named test asserting `{ success: true }` when RPC returns `false`, the distinction silently reverts to `!data` in a refactor and no test catches it. Document the intent inline: "false means already-ended — server decided, client treats it as success." Pattern confirmed: `discard-oral-exam.test.ts` `data === null` guard (2026-07-02).
