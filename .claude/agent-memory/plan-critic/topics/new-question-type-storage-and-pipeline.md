## New question-type storage-shape and pipeline-widening lessons

Relocated verbatim from plan-critic MEMORY.md (curated to stay under the 25 KB native-injection cap). MC-only pipeline generalization gaps and per-item answer storage-shape ambiguity (report pagination vs score-aggregation).

| Plans that generalize an MC-only pipeline to new question types omit the `DraftAnswer` / `AnswerFeedback` type widening AND fail to list `loadSessionQuestions` (the RPC caller) in "Files to change". `DraftAnswer` is `{selectedOptionId,responseTimeMs}` only; `loadSessionQuestions` hard-codes `QuizQuestionRow` to MC columns and silently discards any new RPC output columns. Also: answer recording is batch (`batch_submit_quiz` at session end), not per-question — plans that say "extend the per-question Server Action" for recording are wrong. Round 1 correctly fixed in vfr-rt-training spec (Phase 2.3 lists batch_submit_quiz + DraftAnswer). But: the LATEST `batch_submit_quiz` body (mig 112b/20260619000250) has a HARD `selected_option IS NULL OR ''` RAISE on every answer + an MC option-membership validation loop — so even widening `DraftAnswer` won't help until the migration is extended to branch on question_type. The spec's Phase 2.3 wording "extend batch_submit_quiz to accept and persist response_text/blank_index rows" is correct, but the implementer must know the HARD non-null guard on line 175 must be made conditional, not just additive. | New-type answer storage shape ambiguity: when a new question type requires per-item recording (e.g. `diagram_label` per-zone correctness), the spec must decide BEFORE implementation whether to use (A) single JSON row in response_text (blank_index=NULL) or (B) multiple rows per item (blank_index=zone_index, like dialog_fill). Choice (B) inflates the quiz_session_answers count used for report pagination (count of rows ≠ count of questions), breaking PAGE_SIZE logic. First seen: vfr-rt-training spec Phase 6 `diagram_label`. Plans must specify the storage shape explicitly, not leave it to the implementer. **CONFIRMED 2026-06-21 in the SCORE path too (vfr-rt Phase 2 batch_submit_quiz dispatcher plan):** `batch_submit_quiz` (mig 20260619000250 L239-250) computes `v_answered`/`v_correct_count` via raw `count(*)`/`count(*) FILTER(is_correct)` over quiz_session_answers, then `v_score = v_correct_count / v_answered` (practice) or `/ v_total` (exam). Per-blank dialog_fill rows (choice B) → N rows per question → `count(*)` ≠ question count → score_percentage + answered_count both wrong; passes `db reset` + every MC/short-only test, breaks only on a multi-blank dialog_fill practice session. The CORRECT precedent is `submit_vfr_rt_exam_answers` (mig 20260619000300 L225+) which deliberately uses a `session_questions` CTE for "per-part scores from persisted rows" precisely because raw count() can't be used with per-blank rows. RULE EXTENDED: any plan adding a per-blank/per-item type to an EXISTING grader must also specify the SCORE-aggregation change (DISTINCT question_id, or CTE) + define per-question correctness for partial dialog_fill, AND add a multi-blank-session score integration test — not just the storage shape. Also re-confirmed two sibling batch-grader mechanics for the same plan: (a) non-MC INSERTs must pin `ON CONFLICT (session_id, question_id, blank_index) DO NOTHING` on BOTH quiz_session_answers AND student_responses (the old 2-col UNIQUE was DROPPED in mig 095/20260610000300 → 42P10 if copy-pasting it); (b) the dispatcher's `_batch_questions` TEMP TABLE (L155-164) projects ONLY MC cols → can't supply a `questions`-rowtype/question_type to per-type helpers; plan must widen the projection or switch to a full-row `SELECT q INTO`. Regex strip `\{\{(\d+)\|[^}]*\}\}`→`{{\1}}` is execution-PROVEN in mig 105 (no 2201B; `[^}]` is normal ARE class not POSIX bracket shorthand) — "copy verbatim" is correct de-risking.
