| #781 cookie-rewrite approach for @supabase/ssr: `getUser()` (called in proxy.ts) routes through `_useSession` → `__loadSession` which reads the session from the SSR cookie storage and refreshes only when `expires_at * 1000 - Date.now() < EXPIRY_MARGIN_MS` (= 90 000 ms). The Playwright cookie-rewrite approach (set `expires_at` to a past unix ts) DOES trigger a refresh on the next `getUser()` call because `hasExpired=true` at `__loadSession:2354`. The approach is sound. The fallback (jwt_expiry lowering in config.toml) is a valid secondary option but should not be necessary. Flag: supabase/config.toml's jwt_expiry is a local-dev setting only — CI Supabase in docker uses its own config; lowering it affects the entire CI run, not just the redteam project. Plans proposing this fallback must document the CI blast radius. | 2026-06-07 | 1 | 2026-06-07 | WATCHING |
| E2E un-skip plans for AlertDialogAction with `isPending` text-flip: `AlertDialogAction` is a plain `<Button>` (not a `AlertDialogPrimitive.Close` wrapper). Clicking it fires `onClick` but does NOT dismiss the dialog — the dialog stays open while `isPending=true` (button says "Deleting…") and closes only after the Server Action resolves and the parent re-renders. `name:'Delete'` safely matches the pre-transition text only; the button is disabled during transition so a second click can't race. The dialog stays visible during the transition — plan's `toHaveCount(0)` assertion is only safe AFTER the toast is visible (which is already gated first). Plan was correct on all assumptions; positive signal. | 2026-06-09 | 1 | 2026-06-09 | WATCHING |
| Idempotent-resume RPC race-handler: when wrapping an INSERT in BEGIN...EXCEPTION WHEN unique_violation, a function that is idempotent-resume (returns existing session on race) must do a re-read + RETURN in the handler — NOT mirror the siblings' raise-only shape. start_internal_exam_session (mig 070) and start_exam_session (mig 088) both just RAISE 'active_session_exists' because they are not idempotent. start_vfr_rt_exam_session IS idempotent — the handler must re-read via a SELECT identical to step 5 and return the winner's session. Plans that say "mirror whichever sibling fits best" without explicitly noting this divergence will produce a raise-only handler that breaks idempotency on concurrent first-calls. | 2026-06-10 | 1 | 2026-06-10 | WATCHING |
| Prop-threading plans through intermediate state-holder components must enumerate ALL four file changes: (a) the action/source file returning the new field, (b) the form/trigger component updating its onCallback type, (c) the intermediate state-holder updating its state type and callback lambda, (d) the consumer component adding the prop. Plans that mark the state-holder as "to be confirmed" leave the implementer without a full change list. The test for the form component asserts the exact onCallback shape — if that shape changes, the test breaks. First seen: send-email-code plan (codes-tab.tsx IssuedCode type; issue-code-form.test.tsx line 105 asserts onIssued({code,expiresAt}) which breaks when codeId is added). | 2026-06-18 | 1 | 2026-06-18 | WATCHING |
