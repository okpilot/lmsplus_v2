Recurring traps: (1) **arbitrary-sessionId Server Action soft-delete needs a POSITIVE practice-mode whitelist** — `saveDraft` (draft.ts) takes a client uuid `sessionId` (draft-schema only checks uuid); adding an unguarded `UPDATE quiz_sessions SET deleted_at WHERE id=input.sessionId AND student_id=uid AND ended_at IS NULL` lets a student soft-delete their OWN active internal_exam/vfr_rt_exam → abandon a graded exam (discardQuiz already blocks via `_discard-guard.ts` NON_DISCARDABLE_MODES = internal_exam+vfr_rt_exam; note mock_exam is NOT in it). FIX = `.in('mode',['quick_quiz','smart_review'])`, not the discard blocklist. CRITICAL. (2) **missed callers**: `saveDraft` has 3 client entrypoints all passing sessionId — `saveQuizDraft` (quiz-submit.ts:97), recovery-banner `buildSaveHandler` (quiz-recovery-handlers.ts:54), in-session `useSessionRecovery.handleSave` (use-session-recovery.ts:18) → put the soft-delete in the SERVER action, not one client path, or 2 paths still leak. (3) **resume-via-start_quiz_session mode gate**: RPC whitelists ONLY quick_quiz/smart_review (mig ...600 L46); discovery uses start_discovery_session + can't produce a draft anyway (finish dialog suppressed quiz-finish-dialog-host.tsx:28 + localStorage skipped use-quiz-persistence.ts:12); smart_review not startable → only quick_quiz drafts exist. RPC re-validates every qid vs subject_id/topic_id (L107-118) → stale/deactivated q or wrong stored topic_id (start.ts:53 stores NULL for multi-topic) → invalid_question_ids. PREFER reading mode/subject_id/topic_id from the soft-deleted session row over client-JSONB round-trip. (4) **ordering**: single-active guard forces soft-delete-old BEFORE start; a failing start leaves the draft with no session → surface distinct error tokens (another_session_active vs invalid_question_ids), don't navigate on failure. (5) **JSONB field plumbing**: adding mode/subjectId/topicId to session_config needs Zod(draft-schema)+SaveDraftParsed+sessionConfig()+3 call sites (runner exposes subjectName/Code strings, NOT subjectId); isSessionConfig tolerates extra fields (won't break) but rowToDraftData won't read them. (6) async draft-card handleResume (currently sync) → §6 re-entry ref-guard + loading + await-before-push.
