## Migration-chain and schema-audit plan lessons

Relocated verbatim from plan-critic MEMORY.md (curated to stay under the 25 KB native-injection cap). Plan-review lessons for tracing migration chains, FK/GRANT audits, and distinguishing current-state doc lines from historical delivery records.

| Pattern | First Seen | Count | Last Seen | Status (→ rule loc) |
|---|---|---|---|---|
| Rebase/re-timestamp plans for security PRs with inline `mig NNN` cross-refs inside migration SQL comments: the plan's "targeted edits" scope must enumerate the SQL-file-level comments too (e.g. "Migration 115: … mig 110 … batch_submit_quiz 110b, report RPCs 112/113"), not just docs/ and .claude/rules/. And the `Depends on YYYYMMDD*` timestamp refs inside migration bodies also need re-stamping when files are re-timestamped. Plans that list only docs/ cross-refs miss this. First seen: #856 rebase plan. | 2026-06-19 | 1 | 2026-06-19 | WATCHING |
| Plans that re-number migrations +2 AND add a new spec (count N→N+1) must update spec-count literals in THREE places in tech.md (lines 85, 146, 183) AND in decisions.md — but the PR branch's tech.md may already be stale vs master (branch shows 41, master shows 44). The plan must specify which starting value to use (master's current, not branch's stale). #856 rebase plan: plan says 44→45 but PR856 branch tech.md says 41 (stale from branch cut) while master says 44. | 2026-06-19 | 1 | 2026-06-19 | WATCHING |
| Plans referencing a constraint/mode value cite mig 001 but a later migration widened/renamed it. Must trace the FULL migration chain (grep `ALTER TABLE`, `ON CONFLICT (`, constraint names) before depending on a constraint's shape. | 2026-05-08 | 3 | 2026-05-28 | WATCHING |
| Two-dir migration mirror drift: supabase/migrations can have a newer TIMESTAMP migration that CREATE OR REPLACEs a function without a matching NNN_ packages/db mirror. The result is the same RUNTIME body (supabase controls production) but packages/db's highest NNN file is stale. The plan must identify WHICH directory's latest is binding and reproduce THAT body verbatim. Specifically: `20260521000001_start_quiz_session_null_guard.sql` (supabase) has the NULL guard but has NO packages/db mirror — packages/db 081 is stale. The new 086 migration must base on the supabase latest, not 081. [RETIRED 2026-07-11: packages/db/migrations is FROZEN — supabase/migrations/ is always authoritative; never grep the frozen dir for current SQL. See its README + #1111.] | 2026-06-06 | 1 | 2026-07-11 | RESOLVED (dir frozen 2026-07-11) |
| Sibling-function inline-role-subquery audit: when a plan fixes the inline `(SELECT u.role…)` pattern in one function, sibling SECURITY DEFINER functions with the SAME pattern must be enumerated. The plan's sibling-grep instruction is correct but leaves discovery to the implementer — leaving start_exam_session (054, line 141) unmentioned. This is the same "inline role subquery in audit INSERT" pattern targeted in #734. Fix for start_exam_session is ≤10 lines (add `v_role text` to DECLARE, extend the existing SELECT INTO, replace inline). Plans should name the known offender rather than delegate discovery. | 2026-06-06 | 1 | 2026-06-06 | WATCHING |
| Column GRANT plans on users table: the `users` table has NO explicit column-level GRANT in any migration. `authenticated` holds table-wide UPDATE via Supabase's default schema grants (same mechanism that quiz_sessions used before mig 20260605000001). The REVOKE UPDATE then GRANT UPDATE (full_name) is the correct and non-vacuous shape — mirroring quiz_sessions precedent. The plan's "expected: full_name, possibly updated_at / avatar / locale" is wrong: the users CREATE TABLE has NO updated_at, NO avatar, NO locale columns. The correct re-GRANT is `full_name` only (the only column written by authenticated users via `settings/actions.ts updateDisplayName`). email / deleted_by / last_active_at are only written by service-role or SECURITY DEFINER RPCs. | 2026-06-06 | 1 | 2026-06-06 | WATCHING |
| FK-into-global-table audit: plans that claim "X is the ONLY FK into table Y" must grep ALL migrations for `REFERENCES Y`. For easa_subjects/topics/subtopics: FKs exist in questions (NOT NULL), quiz_sessions (NULLABLE, no ON DELETE clause), and exam_configs (NOT NULL). A plan that only cites questions misses quiz_sessions and exam_configs. However, for test-hermeticity helpers, the practical impact is zero because (a) quiz_sessions.{subject_id,topic_id} are nullable so an easa deletion that left those null would only cause a FK violation on the non-null subjects/topics deleted last, and more importantly (b) cleanupTestData already deletes quiz_sessions before cleanupReferenceData runs — so the FK is satisfied by run order regardless. The plan's stated rationale is wrong but the design is still safe; flag as SUGGESTION not ISSUE. | 2026-06-06 | 1 | 2026-06-06 | WATCHING |
| docs/plan.md Phase-A delivery records vs current-state claims: line 5 is a current-state count ("136 integration tests") — correct bump target. Line 34 is a Phase A delivery record ("136 new SQL tests … Phase A delivered") — bumping it corrupts the historical record. Plans that say "bump lines 5+34" must be corrected to "bump line 5 only." Historical phase-completion entries must not be edited (agent-doc-updater.md rule). | 2026-06-11 | 1 | 2026-06-11 | WATCHING |
