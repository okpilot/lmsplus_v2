## SECURITY DEFINER RPC plan lessons — grants, guards, key-reveal

Relocated verbatim from plan-critic MEMORY.md (curated to stay under the 25 KB native-injection cap). Recurring gaps in plans introducing new SECURITY DEFINER RPCs: false "internal-only" premises, missing per-type correctness guards on key-revealing graders, and omitted SQL conventions.

| Pattern | First Seen | Count | Last Seen | Status (→ rule loc) |
|---|---|---|---|---|
| Plans gate a NEW directly-callable SECURITY DEFINER RPC/helper "internal-only" by merely OMITTING `GRANT EXECUTE TO authenticated` — FALSE PREMISE. Postgres `CREATE FUNCTION` grants EXECUTE to **PUBLIC by default**; `authenticated` inherits via PUBLIC, so an un-granted fn is still callable via PostgREST RPC. To make it internal-only you MUST `REVOKE EXECUTE ... FROM PUBLIC`. VERIFIED vs tree (vfr-rt Phase 2): `grep REVOKE.*EXECUTE.*FROM PUBLIC supabase/migrations/`=**ZERO** — no REVOKE-from-PUBLIC precedent exists. Every directly-callable SECURITY DEFINER RPC is EXPLICITLY `GRANT ... TO authenticated` (check_quiz_answer L127, batch_submit_quiz L306, submit_vfr_rt L300, even pure helper normalize_answer L48 — which is SECURITY INVOKER so harmless). Only un-granted SECURITY DEFINER fns in repo are TRIGGER fns (`sanitize_question_options`, `stamp_last_active` — RETURNS TRIGGER, not RPC-callable) or CREATE-OR-REPLACE inheriting prior grant. CRITICAL when the helper takes a `p_student_id`/`p_session_id` PARAM and writes immutable answer tables (quiz_session_answers/student_responses) with auth/owner/mode guards left ONLY in the dispatcher → cross-user write/forge primitive. FIX (preferred): inline per-type grading in the dispatcher (matches submit_vfr_rt's single-fn CASE-on-type L160-207, zero new surface) OR `REVOKE EXECUTE FROM PUBLIC`+re-derive auth.uid()/ownership in-helper. The §11 sibling-guard rule assumes guards run; a PUBLIC-callable un-guarded helper defeats it. First seen: vfr-rt Phase 2 (3 grade_record_* helpers). **Recurrence (ELP AI-oral-exam Slice 0 grader `write_oral_section_grade`, 2026-07-02): plan explicitly stated "no grant → postgres/service_role only" as the score-forgery closure — FALSE, headline CRITICAL. Two default-grant vectors, not one: (1) Postgres `CREATE FUNCTION` grants EXECUTE to PUBLIC by default; (2) Supabase's initial `ALTER DEFAULT PRIVILEGES … GRANT EXECUTE ON FUNCTIONS TO anon, authenticated, service_role` auto-grants anon+authenticated on every NEW public function regardless of PUBLIC. So omitting the GRANT leaves a student's PostgREST connection able to `POST /rest/v1/rpc/write_oral_section_grade` (SECURITY DEFINER→bypasses RLS→forge scores via caller-supplied `p_descriptor_scores` JSONB). FIX must be explicit `REVOKE EXECUTE ON FUNCTION … FROM PUBLIC, anon, authenticated;` THEN `GRANT … TO service_role;` — REVOKE FROM PUBLIC alone is INSUFFICIENT on Supabase (anon/authenticated hold direct grants). NEW-PATTERN nuance: repo has ZERO `TO service_role`-only definers (grep 2026-07-02) — existing score-writers grant TO authenticated and are safe only because they COMPUTE the score deterministically; a caller-supplied-score grader is a genuinely new surface whose entire safety rests on this REVOKE. Plan's verification DID include a "NOT callable by authenticated" regression test (good) but the stated MECHANISM was wrong so the implementer would not write the REVOKE.** | 2026-06-21 | 2 | 2026-07-02 | WATCHING (CRITICAL when param-driven write to immutable tables; on Supabase must REVOKE FROM PUBLIC **and** anon, authenticated — omission ≠ withholding) |
