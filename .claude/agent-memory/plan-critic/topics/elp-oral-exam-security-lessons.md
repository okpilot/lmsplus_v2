| App-invoke "proxy" of a REVOKE-gated grader re-introduces the forgery the REVOKE prevents: a grader RPC is REVOKEd from PUBLIC/anon/authenticated + GRANTed only to service_role (write_oral_section_grade mig 152 L185-188), and its Edge Function caller authenticates with a shared `x-webhook-secret` + trusts `record.audio_path` from the payload (score-oral-section/index.ts:84-86, downloads the payload path directly, NO DB re-read). A plan that adds an APP-LAYER Server Action to POST to that Edge Function with the secret (ELP Slice-1 score-invoke.ts triggerSectionScoring(responseId,audioPath,sectionNo)) MUST NOT expose it as a directly-callable Server Action: any file in an actions/ dir with `'use server'` makes every export a public POST endpoint, so a client calls triggerSectionScoring(ownResponseId, arbitraryPathUnderOwnPrefix, n) → Edge grades attacker-chosen audio into their real 'grading' response = grade forgery on the assessment (submit RPC's path-prefix + one-path-per-section validation is BYPASSED). FIX: make the invoke a plain non-'use server' server-module helper called only inside submitSectionResponse, AND/OR re-read audio_path from the DB by responseId (ownership-scoped) instead of trusting the caller's path. When a plan says "app-layer invoke of an Edge Function that holds a service-role/secret and forwards a client-supplied resource path," check whether the invoke is a client-reachable Server Action and whether the forwarded path is re-validated server-side.
