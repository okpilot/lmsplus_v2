## Durable knowledge

### Security / RLS / SECURITY DEFINER (highest-value catches)
- **Column-level GRANT pattern for privilege reduction:** when a table's RLS UPDATE policy scopes rows but not columns, the fix is REVOKE the table-level UPDATE then GRANT UPDATE(<safe-col-list>) excluding sensitive columns. Postgres's `SECURITY DEFINER` RPCs run as `postgres` owner and are not subject to the `authenticated` column grant — so scoring/completion RPCs are unaffected. `anon` role gets no explicit table UPDATE grant in this schema, so no `anon` side-channel risk. Verify the column list against all ADD COLUMN migrations when reviewing such patches. Precedent: mig 20260521000004 (internal_exam_codes) + mig 20260605000001 (quiz_sessions).
- **SECURITY DEFINER bypasses RLS** — every SELECT in the body needs manual `AND deleted_at IS NULL` on soft-deletable tables; audit-event INSERT subqueries (actor/session/question FK lookups) need the filter independently of outer guards (security.md §9, §10). Narrow exception: reads by IDs from an immutable write-once column (`quiz_sessions.config.question_ids`). **Cached-role pattern (mig 078, mig 084, 093):** capturing `organization_id + role` in a single deleted_at-filtered read at auth time and reusing the cached variable in later audit INSERTs is §10-compliant — the filter is present at the capture site; no re-query needed. Verify the capture SELECT still includes `AND deleted_at IS NULL`. **Resource-lookup removal exception (mig 093, e3732955):** when a resource existence/org-scope SELECT cannot carry `AND deleted_at IS NULL` because the resource is the just-soft-deleted subject (user.deactivated audit fires after the deactivation), the correct fix is to REMOVE the resource SELECT entirely rather than add a §9-non-compliant filter. The audit row's `organization_id` is always the actor's own org (from the §9-compliant actor lookup), so a bogus/cross-org resource_id is only self-referential log noise — no cross-org read or write. Document the reasoning in a SQL comment. Confirmed: p_resource_id is `UUID NULL, no FK` in audit_events — soft-deleted or cross-org UUIDs are valid label values. Integration test must assert `organization_id === actor's org` to pin the invariant.
- **Multi-permissive SELECT trap:** Postgres ORs permissive policies; a per-caller RPC must self-scope `WHERE owner = auth.uid()` (see tracker). Admin RPCs behind `is_admin()` are exempt.
- **RPCs must self-defend** — Zod at the Server Action boundary is not enough; the RPC is callable directly by any authenticated user. Validate JSONB array length, sum invariants, and UUID element validity inside the function (raw element produces internal `invalid_text_representation`, not a structured `RAISE`).
- **Correct-answer exposure:** student-facing reads must go through `get_quiz_questions()`/report builders that strip `correct` from options. Watch new Server Actions querying `questions.options` directly.
- **Auth-then-validate order:** parse Zod input is fine to throw, but auth check (`auth.uid()`) must gate before any DB work; flag `Schema.parse(input)` placed before the auth check.
- **`.single()` → `.maybeSingle()`** for user/profile lookups where soft-delete yields 0 rows (`.single()` throws PGRST116 as a service error).
- **Trace `CREATE OR REPLACE FUNCTION` to the latest definition** before flagging a missing-pattern finding on a Postgres function (per agent-critic.md Pre-Flag rule).

### Query / data correctness
- **Window-function totals:** `count(*) OVER()` runs before LIMIT → correct full-set total for single-query pagination; but 0-row (out-of-range) page yields no total — caller must treat `totalCount:0` as "maybe out of range".
- **Denominator/numerator mismatch:** widening a read (active-only → non-deleted) for retention UX can push a ratio >100% if the denominator stays narrow — cap with `Math.min(pct,100)` or align (first seen #540).
- **Filter consistency:** count query and page query must carry identical filters + deterministic ordering with a unique tiebreaker (`.order(natural).order('id')`). A single `fetchAllRows` abstraction makes this structural.
- **Zero-row no-op:** DELETE/UPDATE via RLS returns 200 + 0 rows when RLS blocks or filter misses — chain `.select('id')` and check length (code-style.md §5).
- **`as unknown as T` needs a runtime guard** (`Array.isArray`/`typeof`) before use, e.g. `config.question_ids.includes()` throws on a non-array (code-style.md §5).
- **PostgREST embed:** use `!fk_column` (errors loudly) not `:` alias (returns null on FK-resolution failure).
- **camelCase/snake_case JSONB key drift** between TS `.rpc()` payload and SQL `->>'key'` reads is invisible to mocked-client unit tests — needs integration/SQL-parse coverage; prefer snake_case both sides.
- **TOCTOU on COUNT-then-INSERT** (no serializable isolation) is an accepted repo-wide trade-off — document, don't flag as a bug.
- **types.ts nullability:** manually authored RPC `Returns` entries must mirror Postgres nullability — `NUMERIC/TEXT/TIMESTAMPTZ` without NOT NULL → `T | null`.

### React / Next.js correctness
- **Re-throw redirect errors:** any catch wrapping `redirect()`/`notFound()` must `isRedirectError(error)` re-throw (Server Components AND client-component Server Action callers — project has `rethrow-redirect.ts`). Bare `catch{}` turns a redirect into a 500/stale render.
- **Stale-closure on locks/Maps:** deleting from a `lockedRef` in a catch before `setAnswers` propagates opens a double-submit window — release via `useEffect` keyed on the `answers` prop, not synchronously.
- **`new Map()` / `new Set()` in a render return** = new ref each render → downstream dep-array thrash; use a stable `useRef`.
- **Two-layer guard:** HTML `disabled` is not a behavioral contract — `onClick` handlers must also `if (!disabled)` when they manage local state and can be invoked programmatically.
- **`'server-only'` — DO NOT flag its absence (FALSE POSITIVE, #789).** `server-only` is used 0× in this repo; `*/queries.ts` + sibling modules import `adminClient` with no such guard. The earlier "required" claim was wrong — recommending it contradicts the codebase + adds an unused dep.
- **Server Component query helpers** (called via Suspense, not `'use server'`) throwing `error.message` is NOT a client-exposure gap (propagates to error.tsx→Sentry, never reaches DOM) — but a `console.error` before the throw is still wanted for log correlation. Confirmed intentional at count=2; do NOT flag as ISSUE.

### Test-quality catches (semantic, not style)
- **PostgREST NUMERIC(5,2) integer-value serialization:** for integer-valued NUMERIC cells (e.g. `score_percentage = 0` after UPDATE ... SET score_percentage = 0), PostgREST serializes as JSON number `0`, not string `"0.00"`. Playwright `toEqual(0)` passes without `Number()`. Non-integer values (e.g. `66.67`) serialize as strings and need `Number()`. Rule: if the test's DB value is provably always integer (e.g. the expiry branch hardcodes `0`), `toEqual(0)` is safe; if non-integer values are possible in the same column, use `Number()`. Confirmed by rpc-complete-overdue-exam.spec.ts CI passing with `toEqual(0)` direct PostgREST read.
- **Vacuous re-run test:** to verify an effect guard, the dep value passed must actually differ from the initial — same value silently skips the effect, making the assertion vacuous.
- **jsdom flushes `useEffect` in `act()`** — any property depending on an effect appears correct even if the effect fires too late for a real browser; lock-release-after-failure tests pass in jsdom regardless.
- **Mocked-client unit tests can't catch JSONB-key drift or PostgREST string-serialization** — call out the integration-test gap.
- **`vi.importActual` partial mock** (spread real module, override only the fn) keeps exported constants (e.g. `COMMENT_SELECT`) live from source — prefer over full mock with a hardcoded constant string (drift vector).
- **Proxy-based `buildChain` absorbs `.limit()` silently** — `get-active-exam-session.test.ts` + `get-active-internal-exam-session.test.ts` use a `Proxy` that returns itself on any method call. Adding `.limit()` to production code never breaks these tests, but also means no assertion is possible that the cap value is applied. If adding a call-chain assertion to these files, first replace `buildChain` with a per-method `vi.fn()` chain (the `load-draft.test.ts` pattern). Flagged as SUGGESTION on PR #701 (2026-05-30).
- **GoTrue `ban_duration: 'none'` omits `banned_until` from response (undefined, not null):** The SDK `User.banned_until` is typed `string | undefined` (optional field). After an unban, GoTrue omits the field entirely; the JS SDK surfaces `undefined`. The `?? null` coercion maps this to `null` for `.toBeNull()` assertions. A past-date leftover string would also fail the assertion — that failure is correct regression detection behavior. Confirmed PR #372 (ccc1cdf5).
- **Phase-2 skip guard on library contract:** When a phase-2 step is skipped via `if (phase1Result.data.length > 0)`, ensure the skip also covers phase-1 error (not just empty). For `fetchAllRows` callers the two coincide (error → data=[]), but the intent must be explicit with `if (ids.length > 0 && !phase1Result.error)` so a future non-`fetchAllRows` caller doesn't silently send phase-2 with stale/null data. First seen: #684 `collectUserData` session IDs → answers phase-2.
- **`data ?? null` assertion idiom on `RETURNS TABLE` RPCs:** For RPC error paths, `data` is always `null` from PostgREST — `data ?? null` equals `null` and `toBeNull()` passes correctly. But `?? null` is borrowed from scalar-return RPCs where `data: null` on both success and error. On a TABLE-returning RPC's success path `data` would be an array, not null — `?? null` would mask that regression silently. Prefer bare `expect(data).toBeNull()` on error-path assertions for TABLE-returning RPCs. First flagged PR f02031fa #384 (check_consent_status).
- **`expect(error).toBeNull()` on RLS-isolated reads is correct:** PostgREST returns `{ data: [], error: null }` when RLS filters all rows — not an error. Adding `expect(error).toBeNull()` to cross-tenant reads (cross-org, cross-student) distinguishes "RLS blocked → empty array, no error" from "transport/schema failure → error non-null". This is the right assertion pattern on isolation proofs.
- **Empty-array `.in('col', [])` in cleanup helpers:** PostgREST short-circuits `IN ()` to 0 rows without error (Supabase-specific behavior), so it is harmless, but a guard `if (!ids?.length) skip` is more explicit and avoids the implicit behavior. Flag as SUGGESTION when seen in cleanup/setup helpers.

### Process
- **PR-level sweep** (`git diff master...HEAD`) adds value on multi-commit extraction PRs (confirms earlier SUGGESTIONs landed, filter consistency holds cross-commit) but finds nothing extra on single-module refactors.
- **Attack-surface.md commit-ordering stale-state:** when a PR has 2 commits — (1) feature + attack-surface doc written at commit 1, (2) integration test added at commit 2 — the attack-surface rows written at commit 1 may say "GAP" for vectors the commit-2 integration test already covers. The PR-level sweep catches this: the integration test covers the guards, but the doc rows never transitioned to "COVERED AT INTEGRATION LAYER". Fix by updating the doc rows in the same commit that adds the tests. Watch for recurrence (#379, CK/CL/CM/CO).
- **Treat ISSUE as a current gap** regardless of current callers — no "safe today"/"latent" deferral (agent-semantic-reviewer.md).
- **Builder-function dead param:** when extracting inline handlers to `build*` functions that pass all current state as params, a param that was never used in the original (e.g. `setSession` in `buildResumeHandler` — original didn't call it either) should be removed rather than forward-carried. It's noise, and callers passing it incorrectly won't be caught by the type system since `SetState<T>` is structurally identical across all setters. Flag as SUGGESTION on first occurrence; ISSUE if it could mask a missing call.
- **Security doc partial-unique-index threat description:** when documenting an AJ-style "admin bypasses RPC" threat involving a partial unique index, verify whether the index itself closes the threat. If the index covers `WHERE deleted_at IS NULL`, an UPDATE that clears `deleted_at` on a row while a conflicting active row already exists is rejected by the index — the threat vector is narrower than "allows two active rows". Distinguish: (a) index prevents the data-integrity violation entirely vs. (b) only the UX/audit path is bypassed. Mischaracterizing (a) as if the index doesn't help overstates severity.
- **`ensure*` E2E helper unconditional password reset (#593): WON'T-FIX** — GoTrue exposes no hash to compare; a conditional sign-in probe costs ~the same as the idempotent write it would replace, while adding a branch to 4 helpers. The reset is idempotent, does not emit `record_login`/auth events, and Playwright setup runs workers=1 (no lock contention). Do not re-flag.
- **`let refs` uninitialized-ref in afterAll cleanup (#775 pattern):** when `refs` is a describe-scoped `let` assigned in `beforeAll`, a beforeAll throw leaves it `undefined`. afterAll still runs (vitest semantics) and `cleanupReferenceData({ refs: [refs] })` → `.map(r => r.subtopicId)` throws TypeError on `undefined`. This matches the identical gap in `cleanupTestData` (orgId is also an unguarded `let`). Classify SUGGESTION on first occurrence; watch for pattern recurrence. The fix is a null guard: `refs: [refs].filter(Boolean)` or check `if (refs)` before calling. Do not raise to ISSUE: (a) it is only exposed during beforeAll crashes (not normal runs), (b) the suite already fails loudly in that scenario, and (c) the pattern is pre-existing across all 7 suites in the same file.
